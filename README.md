## GPU Kernel Profiler

![GPU Kernel Profiler UI](demo/ProfilerDemo.png)

The **GPU Kernel Profiler** is our custom web-based dashboard designed to rapidly evaluate and visualize Helion kernel performance during the hackathon.

### How it Works

* **Trigger Experiments:** Use the "Deploy to B200" panel in the UI to remotely dispatch jobs. You can select the kernel (e.g., `causal_conv1d`, `fp8_quant`) and the execution mode (`benchmark`, `profile`, or `test`). The web app executes scripts to deploy the job directly to our Nebius B200 instance.
* **Retrieve Results:** The dashboard continuously polls for new logs generated by the remote instance. Once a job is complete, the logs are synced back to the local `helion/logs` directory.
* **Visualize Performance:** The UI parses the raw `.log` files (including PyTorch Profiler tables) and generates interactive bar charts using Chart.js.
* **Identify Optimal Configs:** The table view lists all tested shape configurations, extracts their execution time, identifies the percentage of time spent on the GPU, and automatically highlights the absolute fastest configuration (★) so you can hardcode it into `submission.py`.

## Kernel Optimizations Summary

We implemented several high-level algorithmic and hardware-specific optimizations across the Helion kernels to maximize throughput on the B200:

### 1. Mathematical Factorization (The "Exp" Trick)
To avoid the `O(T^2)` memory bottleneck of computing large 2D exponential gate matrices, we factorized the gated attention rule:
* **Original:** `sim_{ij} = exp(g_i - g_j)`
* **Optimized:** `exp(g_i) * exp(-g_j)`
* **Result:** We scale `Q` by `exp(g_i)` and `K` by `exp(-g_j)` **before** the dot product. This transforms a quadratic memory operation into two linear-time vector scales, allowing us to keep everything in SRAM and avoid HBM spills.

### 2. SRAM Tiling & Block-Level Parallelism
We utilized `hl.tile` to map computations to the B200’s massive SRAM:
* **Chunking:** Kernels process data in blocks (typically `C=64`), allowing us to load a tile of `Q`, `K`, `V` once and perform both local (intra-chunk) and global (inter-chunk) attention within a single loop.
* **Vectorization:** We ensured data was loaded using vectorized patterns (e.g., matching the 64-element block size) to maximize the B200's memory bus utilization.

### 3. Fused Recurrence State Updates
In the Gated DeltaNet kernels (like `chunk_fwd_h`), we fused the update of the hidden state `H` directly into the processing loop:
* Instead of computing `H` as a separate global pass, we calculate the update `W` and `U` and apply them to the state `H` within the kernel. This significantly reduces the overhead of reading and writing the large `H` state matrix to HBM.

### 4. Hardware-Specific Configurations (ACF & Autotuning)
We moved away from generic defaults to hardcoded, B200-optimized configurations:
* **Advanced Controls (ACF):** We applied custom `.acf` files provided by the `booster_pack` to tune instruction scheduling and pipeline depth.
* **Warp & Stage Tuning:** Through remote benchmarking, we identified that `num_warps=8` and `num_stages=2` provided the best balance of compute density and latency for the B200's architecture.

### 5. Lazy Kernel Compilation
To prevent the Python environment from hanging or timing out during initialization on the remote instance:
* We implemented a **Lazy Initialization** pattern. Kernels are only compiled upon the first request for a specific tensor shape, ensuring the submission script starts instantly.

### 6. Causal Masking via Arange
Instead of passing a large boolean mask from PyTorch, we generate the causal mask dynamically inside the kernel using `hl.arange(tile_t.block_size)` and `torch.where`. This saves significant HBM bandwidth and leverages the GPU's register-level logic.
